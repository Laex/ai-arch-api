### 1. Подготовительный этап и интеграция MLflow

- **Добавить `mlflow` в файл `pyproject.toml` и установить зависимости.**

	```ini
	[tool.poetry.dependencies]
	mlflow = "^2.11.0"
	```

Обновляем Poetry (скачать и установить новые зависимости в виртуальное окружение):

```bash
	poetry update
```

Либо, как альтернативный вариант, можно просто выполнить команду `poetry add mlflow` в терминале – это автоматически добавило бы актуальную версию библиотеки в файл `pyproject.toml` и сразу установило бы её.

- **Модифицировать `docker-compose.yml`:**
	- Добавить новый сервис `mlflow`, используя образ `ghcr.io/mlflow/mlflow`.
	- Настроить `volumes` для `mlflow` для хранения артефактов (например, `./mlflow_artifacts:/mlflow/artifacts`).
	- Задать `command` для запуска `mlflow` сервера.
	- В сервисах `api` и `worker` добавить `depends_on: [mlflow]` и переменную окружения `MLFLOW_TRACKING_URI`.

**В итоге:**

1. Добавлен сервис `mlflow`:
	- Используется официальный образ `ghcr.io/mlflow/mlflow`.
	- Веб-интерфейс будет доступен на `http://localhost:5000`.
	- В `command` задан запуск сервера, который будет хранить метаданные экспериментов в файле `mlflow.db` внутри контейнера, а артефакты (модели, метрики) – в папке `/mlflow/artifacts`.
2. Настроены `volumes`:
	- Локальная папка `./mlflow_artifacts` будет создана на компьютере и подключена к контейнеру. В ней можно будет найти все артефакты, которые будут логироваться в MLflow.
3. Обновлены сервисы `api` и `worker`:
	- Добавлена переменная окружения `MLFLOW_TRACKING_URI=http://mlflow:5000`. Теперь Python-скрипты внутри этих контейнеров будут знать, куда отправлять данные для логирования.
	- Указана зависимость `depends_on: [mlflow]`, чтобы `api` и `worker` запускались только после старта MLflow-сервера.

Теперь можно протестировать и запустить всю систему командой `docker-compose up --build`, поднимется еще один сервис – MLflow Tracking Server.

- **Модификация скрипта обучения (`scripts/train_model.py`):**
	- Настроить подключение к MLflow, используя `MLFLOW_TRACKING_URI`.
	- Установить имя эксперимента.
	- Обернуть логику обучения в `with mlflow.start_run():`.
	- Логировать параметры (`mlflow.log_params`), метрики (`mlflow.log_metric`) и модель (`mlflow.onnx.log_model`), зарегистрировав её под именем, соответствующим вашему варианту (например, `scoring_model`).

	`scripts/train_model.py`, файл приведен полностью, так как изменения большие.

	В код добавлена инициализация MLflow: теперь скрипт подключается к серверу (через `MLFLOW_TRACKING_URI`), создает эксперимент, а вся логика обернута в блок `with mlflow.start_run()`. Также добавлено логирование параметров пайплайна, базовой метрики (accuracy) и регистрация сконвертированной ONNX-модели в Model Registry под именем `document_classifier`. Старое локальное сохранение в папку заменено логированием в MLflow.

```python
import os
import mlflow
import mlflow.onnx
from sklearn.feature_extraction.text import TfidfVectorizer
from sklearn.linear_model import LogisticRegression
from sklearn.pipeline import Pipeline
from skl2onnx import convert_sklearn
from skl2onnx.common.data_types import StringTensorType

# Настройка подключения к MLflow
MLFLOW_TRACKING_URI = os.getenv("MLFLOW_TRACKING_URI", "http://localhost:5000")
mlflow.set_tracking_uri(MLFLOW_TRACKING_URI)
mlflow.set_experiment("Document Classification")

def train():
    # Оборачиваем логику в запуск MLflow
    with mlflow.start_run() as run:
        # 1. Подготовка данных
        X = [
            "Счет на оплату услуг №123", 
            "Оплата по договору", 
            "Сумма к оплате 500р", 
            "Выставить счет фактуру",
            "Договор аренды помещения", 
            "Соглашение о конфиденциальности", 
            "Подписи сторон",
            "Дополнительное соглашение к договору"
        ]
        y = ["invoice", "invoice", "invoice", "invoice", "contract", "contract", "contract", "contract"]

        # 2. Создание и обучение пайплайна
        params = {"tfidf__max_features": 100, "clf__C": 1.0}
        pipeline = Pipeline([
            ('tfidf', TfidfVectorizer(max_features=params["tfidf__max_features"])),
            ('clf', LogisticRegression(C=params["clf__C"]))
        ])

        print("Обучение модели...")
        pipeline.fit(X, y)

        # Логируем параметры и метрики в MLflow
        mlflow.log_params(params)
        accuracy = pipeline.score(X, y)
        mlflow.log_metric("accuracy", accuracy)

        # Тестовое предсказание
        test_text = "Прошу оплатить счет"
        print(f"Тест: '{test_text}' -> {pipeline.predict([test_text])[0]}")

        # 3. Конвертация в ONNX
        initial_type = [('input_text', StringTensorType([None, 1]))]
        print("Конвертация в ONNX...")
        onnx_model = convert_sklearn(pipeline, initial_types=initial_type)

        # 4. Сохранение (логирование) модели в MLflow Model Registry
        print("Логирование модели в MLflow...")
        mlflow.onnx.log_model(
            onnx_model,
            artifact_path="model",
            registered_model_name="document_classifier"
        )

        print(f"Обучение завершено. Модель залогирована в MLflow! Run ID: {run.info.run_id}")

if __name__ == "__main__":
    train()
```

### 2. Рефакторинг сервиса для загрузки модели из MLflow
- Изменить `InferenceService` (фабрику для его создания в `src/presentation/dependencies.py`).
- Реализовать логику, которая будет загружать модель из MLflow Model Registry.
- Использовать URI модели, например, `models:/scoring_model/Production`) для загрузки нужной версии.

Изменяем фабрику зависимостей `get_model()` в слое Presentation. Вместо загрузки локального `.onnx` файла система будет обращаться к MLflow Model Registry, скачивать последнюю версию модели, находящуюся в стадии `Production`, и инициализировать классификатор из неё. На случай, если MLflow недоступен можно оставить "резервный" вариант загрузки локального файла.

Измененный `src/presentation/dependencies.py`

```python
import os
import mlflow
from functools import lru_cache
from fastapi import Depends
from src.infrastructure.onnx_model import ONNXDocumentClassifier
from src.application.services import DocumentRoutingService

# Подключение к MLflow
MLFLOW_TRACKING_URI = os.getenv("MLFLOW_TRACKING_URI", "http://localhost:5000")
mlflow.set_tracking_uri(MLFLOW_TRACKING_URI)

@lru_cache(maxsize=1)
def get_model():
    """
    Загружает модель в память (Singleton).
    Используется lru_cache, чтобы не создавать объект модели заново на каждый запрос.
    """
    model_name = "document_classifier"
    model_alias = "production"
    model_uri = f"models:/{model_name}@{model_alias}"
    
    print(f"[Model] Загрузка модели из MLflow Registry ({model_uri})...")
    try:
        # MLflow автоматически найдет версию модели с указанным псевдонимом (alias) и скачает её
        model_dir = mlflow.artifacts.download_artifacts(artifact_uri=model_uri)
        model_path = os.path.join(model_dir, "model.onnx")
    except Exception as e:
        print(f"[Model] Ошибка MLflow: {e}. Загрузка локальной копии...")
        model_path = "models/classifier.onnx"
        
    return ONNXDocumentClassifier(model_path)

def get_service(classifier: ONNXDocumentClassifier = Depends(get_model)) -> DocumentRoutingService:
    """
    DI-провайдер для сервиса бизнес-логики.
    FastAPI автоматически внедрит сюда результат get_model.
    """
    # Здесь происходит сборка (Composition Root) для каждого запроса,
    # но тяжелый объект модели (classifier) создается только один раз благодаря lru_cache
    return DocumentRoutingService(classifier=classifier)
```

**Внимание! На этом этапе много чего работает - вручную!**

В веб-интерфейсе MLflow:
1. **Models** -> **document_classifier** -> выбрать версию модели (например, Version 1).
2. В секции **Aliases** добавить (или редактировать) - псевдоним `production`.
Воркер и API будут обращаться по URI `models:/document_classifier@production` и без предупреждений скачивать нужную версию!

### 3. Создание`CI/CD`пайплайна

Чтобы пайплайн работал корректно, нам нужно выполнить три вещи:
* Добавить линтер ruff в зависимости проекта для тестирования качества кода.

```ini
[tool.poetry.group.dev.dependencies]
...
ruff = "^0.3.0"
```

* Создать скрипт Quality Gate (`scripts/quality_gate.py`), который проверит, что метрика (accuracy) обученной модели удовлетворяет заданию (в этом примере - `accuracy` > 0.90).

`scripts/quality_gate.py` - в случае, если точность (accuracy) на тестовых данных будет менее `0.90`, пайплайн выдаст код ошибки `1`, завершит выполнение, и до стадий `train` и `build` дело не дойдет

```python
import sys
from sklearn.feature_extraction.text import TfidfVectorizer
from sklearn.linear_model import LogisticRegression
from sklearn.pipeline import Pipeline

def check_accuracy():
    print("Запуск Quality Gate: проверка базовой метрики модели...")
    
    # 1. Подготовка данных для тестирования (аналогичные обучающим)
    X = [
        "Счет на оплату услуг №123", 
        "Оплата по договору", 
        "Сумма к оплате 500р", 
        "Выставить счет фактуру",
        "Договор аренды помещения", 
        "Соглашение о конфиденциальности", 
        "Подписи сторон",
        "Дополнительное соглашение к договору"
    ]
    y = ["invoice", "invoice", "invoice", "invoice", "contract", "contract", "contract", "contract"]

    pipeline = Pipeline([
        ('tfidf', TfidfVectorizer(max_features=100)),
        ('clf', LogisticRegression(C=1.0))
    ])
    pipeline.fit(X, y)
    
    accuracy = pipeline.score(X, y)
    print(f"Текущая метрика (accuracy): {accuracy:.2f}")
    
    if accuracy <= 0.90:
        print(f"[Error] Quality gate FAILED: accuracy {accuracy:.2f} не превышает порог 0.90")
        sys.exit(1)
    else:
        print("[Success] Quality gate PASSED!")

if __name__ == "__main__":
    check_accuracy()

```

Создать сам файл конфигурации пайплайна `ci.yml` (для GinHub) или `.gitlab-ci.yml`(для GitLab - копайте документацию сами) в корне проекта и определить в нём три стадии: `test`, `train`, `build`. Конфигурационные файлы для GitHub должны лежать в папке `.github/workflows/`.

Файл `.github/workflows/ci.yml`

```yaml
name: AI System CI/CD

# Запускать пайплайн при пуше в ветку main/master и при создании Pull Request
on:
  push:
    branches:
      - main
      - master
  pull_request:
    branches:
      - main
      - master

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Set up Python 3.11
        uses: actions/setup-python@v5
        with:
          python-version: '3.11'

      - name: Install Poetry
        run: |
          pip install poetry
          poetry config virtualenvs.create false

      - name: Install dependencies
        run: poetry install --no-interaction --no-ansi

      - name: Run linters
        run: poetry run ruff check .

      - name: Check model quality gate
        run: poetry run python scripts/quality_gate.py

  train:
    runs-on: ubuntu-latest
    needs: test
    if: github.ref == 'refs/heads/main' || github.ref == 'refs/heads/master'
    env:
      # Используем localhost, так как MLflow будет поднят прямо на раннере
      MLFLOW_TRACKING_URI: "http://localhost:5000"
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Start MLflow server for testing
        run: docker-compose up -d mlflow

      - name: Set up Python and Poetry
        uses: actions/setup-python@v5
        with:
          python-version: '3.11'
      - run: |
          pip install poetry
          poetry config virtualenvs.create false
          poetry install --no-interaction --no-ansi

      - name: Train model and log to MLflow
        run: poetry run python scripts/train_model.py

  build:
    runs-on: ubuntu-latest
    needs: train
    if: github.ref == 'refs/heads/main' || github.ref == 'refs/heads/master'
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Log in to GitHub Container Registry
        run: echo "${{ secrets.GITHUB_TOKEN }}" | docker login ghcr.io -u ${{ github.actor }} --password-stdin

      - name: Build and push API and Worker images
        run: |
          # Собираем и пушим образы в GitHub Container Registry (ghcr.io)
          IMAGE_ID=ghcr.io/${{ github.repository }}
          # Приводим имя репозитория к нижнему регистру (требование Docker)
          IMAGE_ID=$(echo $IMAGE_ID | tr '[A-Z]' '[a-z]')
          
          docker build -f Dockerfile.api -t $IMAGE_ID/api:latest .
          docker push $IMAGE_ID/api:latest
          
          docker build -f Dockerfile.worker -t $IMAGE_ID/worker:latest .
          docker push $IMAGE_ID/worker:latest
```

**Внимание!**

1. Перед отправкой в репозиторий, естественно, необходимо выполнить команду `poetry update`, чтобы Poetry обновил файл блокировок (`poetry.lock`) из-за того, что добавлен пакет `ruff` в `pyproject.toml`.
2. Файл называется `ci.yml` и должен находиться строго в директории `.github/workflows/`.
3. Используется GitHub Container Registry (`ghcr.io`). Для авторизации используется автоматически генерируемый токен `${{ secrets.GITHUB_TOKEN }}`, не нужно настраивать его вручную.
4. GitHub Actions использует готовые экшены (например, `actions/setup-python@v5` и `actions/checkout@v4`), которые берут на себя задачу по загрузке репозитория и настройке Python на виртуальной машине.

Где загрузка модели?

В FastAPI зависимости, внедряемые через `Depends()` (`Depends(get_model)`), вычисляются **лениво** (lazy loading). Это означает, что функция `get_model()` не запускается в момент старта самого сервера (`Application startup complete`). Она будет вызвана только тогда, когда в API поступит **первый реальный запрос** к эндпоинту, который требует эту зависимость.

Благодаря декоратору `@lru_cache(maxsize=1)` над функцией `get_model()`, произойдет следующее:
- При **первом** запросе пользователя (например, при отправке текста на классификацию) FastAPI вызовет функцию `get_model()`.
- Именно в этот момент в логах появится строка `[Model] Загрузка модели из MLflow Registry...`.
- Модель скачается из MLflow и загрузится в память.
- При **всех последующих** запросах функция больше вызываться не будет – FastAPI просто возьмёт уже готовую модель из кэша.

### 5. Очистка проекта

В проекте присутствуют логические "рудименты" (неиспользуемые зависимости и куски инфраструктуры), которые остались от начальных этапов проектирования (до полного перехода на MLflow). Надо навести чистоту.

Что можно безболезненно убрать:
1. Сервис MinIO в Docker
	В `docker-compose.yml` всё ещё работает тяжелый сервис объектного хранилища minio, а в контейнеры api и worker пробрасываются переменные MINIO_ENDPOINT, MINIO_ACCESS_KEY и т.д. Поскольку MLflow теперь работает в режиме Artifact Proxy (см. `docker-compose.yml`) и сохраняет модели в локальную папку `./mlflow_artifacts`, а датасет в `train_model.py` жестко задан в памяти (массивы X и y), хранилище `MinIO` простаивает вхолостую. Можете удалить блок `minio:` из файла `docker-compose`, а также его том `minio_storage:`, чтобы сэкономить ресурсы.

2. Зависимости DVC и S3
	В файле `pyproject.toml` установлены библиотеки `boto3`, `dvc` и `dvc-s3`. Обычно они используются связке с `MinIO` для получения моделей (`DVC pull`). Так как теперь API обращается напрямую к `mlflow.artifacts.download_artifacts`, эти зависимости ненужны. Их можно удалить командой:

```bash
poetry remove dvc dvc-s3 boto3
```

Оставлять `MinIO` и `DVC` имеет смысл только в том случае, если на следующих этапах планируетcя хранить там сырые текстовые документы для обучения, но для управления именно моделями они больше не нужны.

- `Dockerfile.api` и `Dockerfile.worker` – необходимы (используются на шаге build).

```python
"""
Модуль API (Presentation Layer) на базе FastAPI.

Этот модуль отвечает за обработку HTTP-запросов. Он является точкой входа
для внешних систем (веб-фронтенд, мобильные приложения, другие сервисы).
Здесь происходит преобразование HTTP-запросов в вызовы бизнес-логики (Application Layer).
"""
from contextlib import asynccontextmanager
from fastapi import FastAPI
from pydantic import BaseModel
from src.presentation.dependencies import get_model
from src.presentation.tasks import predict_task
from src.presentation.celery_app import celery_app

@asynccontextmanager
async def lifespan(app: FastAPI):
    """
    События жизненного цикла приложения (Lifespan Events).
    
    Этот менеджер контекста позволяет выполнять код:
    1. При запуске приложения (до начала обработки запросов) - блок до yield.
    2. При остановке приложения - блок после yield.
    
    Здесь мы используем это для предварительной загрузки модели в память,
    чтобы она скачалась из MLflow до того, как придет первый пользовательский запрос.
    """
    print("[Startup] Инициализация приложения...")
    try:
        # Предзагрузка модели в память, чтобы первый запрос не тормозил
        get_model()
    except Exception as e:
        print(f"[Startup Error] Ошибка инициализации: {e}")
    
    yield
    # --- 2. Shutdown (Очистка ресурсов) ---
    # Здесь можно закрыть соединения с БД или остановить фоновые задачи.
    print("[Shutdown] Остановка сервера...")

app = FastAPI(title="Document Classifier API", lifespan=lifespan)

# DTO (Data Transfer Object) для запроса
class DocumentRequest(BaseModel):
    """
    Модель данных для входящего JSON-запроса.
    FastAPI использует её для валидации входных данных.
    """
    filename: str
    content: str

class TaskResponse(BaseModel):
    task_id: str

@app.get("/")
def root():
    """
    Проверка работоспособности API (Health Check).
    Используется системами мониторинга (k8s probes) для проверки статуса сервиса.
    """
    return {"message": "API is running. Go to /docs to test the classifier."}

@app.get("/classify")
def classify_info():
    """
    Подсказка, если пользователь случайно открыл этот адрес в браузере (GET-запрос).
    Браузеры по умолчанию делают GET, а наш метод требует POST.
    """
    return {"message": "GET method not allowed. Please use POST request with JSON body to classify documents. Check /docs for details."}

@app.post("/classify", response_model=TaskResponse, status_code=202)
def classify_document(request: DocumentRequest):
    """
    Эндпоинт для классификации документа (Асинхронный).
    
    Принимает JSON с текстом документа, создает задачу в Celery и возвращает ID задачи.
    """
    # 1. Отправка задачи в очередь (Redis)
    task = predict_task.delay(
        filename=request.filename,
        content=request.content
    )
    
    # 2. Возврат ID задачи клиенту
    return {"task_id": task.id}

@app.get("/classify/result/{task_id}")
def get_task_result(task_id: str):
    """
    Эндпоинт для проверки статуса задачи и получения результата.
    """
    task_result = celery_app.AsyncResult(task_id)
    
    if task_result.ready():
        if task_result.successful():
            return {
                "task_id": task_id,
                "status": "SUCCESS",
                "result": task_result.get()
            }
        return {
            "task_id": task_id,
            "status": "FAILURE",
            "error": str(task_result.result)
        }
    
    return {"task_id": task_id, "status": task_result.status}

class DocumentRoutingService:
    """
    Сервис (Use Case) для маршрутизации документов.
    """

    def __init__(self, classifier: IDocumentClassifier):
        """
        Инициализация сервиса с внедрением зависимостей.
        """
        self.classifier = classifier

    def preprocess(self, text: str) -> str:
        """
        Обрезает текст до 1000 символов для оптимизации.
        """
        return text[:1000]

    def run(self, filename: str, raw_content: str) -> DocCategory:
        """
        Выполнение бизнес-сценария классификации.
        """
        # 1. Предобработка
        clean_content = self.preprocess(raw_content)

        # 2. Создание сущности
        doc = DocumentText(filename=filename, content=clean_content)
        
        # 3. Вызов модели
        return self.classifier.classify(doc)
```

В кодовой базе остался один файл, который сломается при запуске, потому что он всё ещё пытается импортировать удаленные классы (`S3Storage` и `DataSyncService`). Это скрипт интерфейса командной строки — `cli.py`. Надо использовать загрузку моделей через MLflow (через `dependencies.py`).

`src/presentation/cli.py`
```python
"""
Модуль интерфейса командной строки (CLI) слоя Presentation.

Этот слой отвечает за взаимодействие с внешним миром (пользователем).
Его задачи:
1. Принять ввод от пользователя (аргументы командной строки).
2. Собрать приложение (Composition Root): инициализировать инфраструктуру и внедрить её в бизнес-логику.
3. Запустить сценарий использования (Use Case).
4. Вывести результат в понятном пользователю формате.
"""
import sys
from src.presentation.dependencies import get_model, get_service

def main():
    """
    Точка входа в приложение.
    
    Здесь происходит сборка зависимостей (Dependency Injection).
    Слой Presentation знает обо всех остальных слоях, чтобы соединить их вместе.
    """
    # --- 1. Инициализация (Composition Root) ---
    print("[CLI] Инициализация сервисов и загрузка модели...")
    classifier = get_model()
    service = get_service(classifier=classifier)
    
    # --- 2. Взаимодействие с пользователем ---
    # Получение данных из внешнего мира (аргументы CLI)
    input_text = "Hello World"
    if len(sys.argv) > 1:
        input_text = sys.argv[1]
        
    print(f"Running inference on: '{input_text}'")
    
    # --- 3. Запуск основного сценария использования ---
    # Для CLI используем фиктивное имя файла
    result = service.run(filename="cli_input.txt", raw_content=input_text)
    
    # --- 4. Вывод результата ---
    print(f"Result: {result.label} (Confidence: {result.confidence})")

if __name__ == "__main__":
    main()
```

>Сделать коммит и пуш изменений в GitHub репозиторий. Проверить в GitHub UI, что пайплайн успешно выполнился. Проверить в веб-интерфейсе MLflow, что появился новый запуск с метриками и моделью. Если не сделали автоматическое назначения алиаса - вручную изменить стадию новой модели на `Production` или `Staging` в MLflow. Запустить систему через `docker-compose up` и убедиться, что `worker` загружает новую модель из MLflow и система корректно обрабатывает запросы.

Запуск пайплайна в GitHub Actions происходит **автоматически**, как пушится изменения в ветку `main` или `master`. Это связано с тем, что в файле `.github/workflows/ci.yml` прописан блок `on: push: branches: [main, master]`.

1. Зафиксировать все изменения
	Убедится, что находимся в корне проекта. Отправить все новые файлы (скрипты, изменённый `docker-compose.yml`, папку `.github` и обновлённый `poetry.lock`) в репозиторий. Из терминала:
```bash
git add . 
git commit -m "feat: Интеграция MLflow и настройка CI/CD через GitHub Actions"
git push origin main  
# git push origin master
```

2. Открыть вкладку `Actions` в `GitHub` - перейти на страницу репозитория на сайте `GitHub`, нажать на вкладку `Actions`, увидеть запущенный процесс с названием `AI System CI/CD` (смотри `ci.yml`), посмотреть процесс, увидеть детали.

3. Посмотреть за стадиями (Jobs)
	Визуальный граф из трёх квадратиков (наших стадий), которые будут выполняться последовательно:
	- **test:** Сначала GitHub поднимет чистую виртуальную машину (Ubuntu), установит Python, Poetry, скачает зависимости проекта и запустит `ruff` (линтер). Затем он запустит скрипт `scripts/quality_gate.py`. Если `accuracy` будет выше `0.90`, шаг загорится зелёным.
	- **train:** Как только завершится `test` - внутри облачного раннера GitHub поднимется Docker-контейнер `MLflow`. Затем запустится `scripts/train_model.py`, модель обучится, сохранится в этот временный `MLflow`, и ей автоматически назначится алиас `production`.
	- **build:** Авторизация в реестре GitHub (GitHub Container Registry) с помощью встроенного токена, собор двух образов (`Dockerfile.api` и `Dockerfile.worker`), отправка их в облако.

4. Проверка сохраненных Docker-образов
	Когда пайплайн полностью загорится зелёным, необходимо вернуться на главную страницу репозитория на GitHub. Справа - раздел `Packages`. Там появятся свежесобранные образы `api` и `worker`. Теперь любой сервер сможет загрузить их и развернуть готовую Production-систему. Если на каком-то этапе возникнет ошибка (красный крестик), можно кликнуть на упавший шаг – GitHub покажет подробный консольный лог, где будет понятно, что пошло не так (например, опечатка в коде или упавший Quality Gate).
